HTDemucs model core

The inference pipeline page treats the model as a black box: f32 (1, 2, 343980) in, f32 (1, 4, 2, 343980) out. This page opens that box. The architecture is HTDemucs (demucs v4): the same 7.8 s of audio is processed twice in parallel — once as a spectrogram, once as a raw waveform — by two convolutional U-Nets that never touch in the encoder and exchange information only through a transformer at the bottleneck (§4), which is also where 75% of the weights live (§6). Paper: arXiv:2211.08553 · code: htdemucs.py (the pinned fork this repo exports from) · upstream mechanics notes: demucs.md.

1 · One idea: separate in two views at once

Sound has two natural representations, and each makes a different kind of structure easy to see. Earlier Demucs generations picked one (v1/v2 waveform-only, spectrogram models like Open-Unmix the other); v3 ran both; v4 (this model) runs both and lets them talk.

Spectrogram view — frequency × time grid

  • A sustained bass note or vocal harmonic is a few stable horizontal rows — trivially separable structure
  • A drum hit is smeared vertically across all 2048 bins — hard
  • Phase (needed to get a waveform back) is the classically hard part; §5 shows how this model sidesteps masking entirely

Waveform view — raw samples

  • A drum transient is a sharp local event — exactly what a dilated conv stack resolves; phase comes for free
  • Pitch and harmony are spread over thousands of samples — hard
  • Waveform U-Nets (demucs v1/v2) historically won on drums/bass, spectrogram models on vocals/other

Each view's blind spot is the other's strength. The model exploits that literally: the final output is the sum of two independent waveform estimates, one reconstructed from the spectrogram branch and one produced directly by the time branch (htdemucs.py:682, x = xt + x).

2 · The shape: two U-Nets and a bridge

Both branches are 4-layer conv encoder/decoder pyramids with skip connections (U-Nets). Each encoder layer trades resolution for channels: ÷4 resolution, ×2 channels. The spectrogram branch strides along the frequency axis only (time frames pass through untouched); the time branch strides along time. After 4 layers both arrive at 384 channels, the transformer bridges them, and mirrored decoders climb back up using the skips. With this depth the two branches never merge in the encoder — the merge path in the code (inject) is dead for depth 4 / nfft 4096, so the bridge is the only place the views meet.

Reading the shapes

  • Tensors are (batch, channels, …space) — batch is always 1; channels are learned feature maps
  • Spec branch: (1, C, freq, 336) — 336 = STFT frames of 7.8 s at hop 1024
  • Time branch: (1, C, samples)

Shapes measured by tracing the shipped htdemucs checkpoint (they are identical for all five weight sets). The odd time-branch lengths (85995 → 21499 → 5375) are just ÷4 with a right-pad when not divisible. Forward pass: htdemucs.py:551–685.

3 · Inside one encoder layer

All eight encoder layers (both branches) are the same recipe, HEncLayer: a big strided conv does the down-sampling, then a small residual conv stack (DConv) refines, then a gated 1×1 mixes channels.

  • The dilated convs always run along time — in spectrogram layers the tensor is reshaped so every frequency band becomes its own batch item (hdemucs.py:146). That is how "frequency" layers still see temporal context.
  • Frequency embedding (spec branch, after layer 0 only): the strided conv shares weights across all 512 bins, so a learned 48-dim vector per bin is added to tell the network which pitch region it is looking at (htdemucs.py:601).
  • Decoder layers (HDecLayer) mirror this without the DConv: add the skip, gated 3-wide conv, then a transposed conv does the ×4 upsample.

4 · The bridge: where the two views talk

At the bottleneck each branch is compact enough to attend over. Both are projected 384→512 and flattened into token sequences (transformer.py:648):

2688 spectrogram tokens

  • 8 freq bands × 336 frames, one 512-dim token each
  • 2-D sinusoidal position embedding (freq, time)

1344 waveform tokens

  • one 512-dim token per 256 samples (~5.8 ms)
  • 1-D sinusoidal position embedding

Then 5 layers alternate two moves. Self-attention layers (0, 2, 4) let each branch reorganize globally on its own — note that this is the first time the time branch can relate sample 85 to sample 340000 directly, since convs are local. Cross-attention layers (1, 3) are the hybrid payoff: each spectrogram token builds its update as an attention-weighted mix of the waveform branch's tokens, and vice versa (queries from one branch, keys/values from the other, computed symmetrically from the same pre-layer states). Concretely, the token holding a smeared drum transient in the spectrogram can pull the crisp version from the time branch, and the time token in a dense chord can pull the resolved harmonics from the spectrogram side. Each of the 10 layer instances (5 per branch) has its own weights: dim 512, 8 heads, FFN 2048, pre-LayerNorm, LayerScale.

Why this block dominates the model

  • a self-attn instance ≈ 3.15 M params (QKV+out ≈ 1.05 M, FFN ≈ 2.1 M); a cross-attn instance is the same size
  • 10 instances + norms ≈ 31.6 M of the 42.0 M total (75%), measured, §6
  • attention cost stays sane because it runs on 2688 / 1344 tokens, not on raw resolution — the conv pyramids exist to make the transformer affordable

5 · Rebuild: no mask, just predict and sum

Spectrogram separators traditionally output a mask over the mix's magnitudes and reuse the mix's phase. This model does something simpler and stronger — complex-as-channels (CaC): real and imaginary parts are just 2 channels per audio channel, end to end.

  • Input side: the mix STFT becomes 4 channels (2 stereo × re/im).
  • Output side: decoder.3 emits 16 channels = 4 sources × 2 stereo × re/im — a full complex spectrogram per stem, phase included, predicted directly. The "masking" function is a pure reshape when CaC is on (htdemucs.py:483).
  • iSTFT turns that into waveforms, and the time branch's own 8-channel output (4 sources × stereo) is added: output = iSTFT(spec_est) + time_est. Each branch alone would already be a complete separator; the sum is an ensemble baked into one network, trained jointly.
  • The in-graph z-score normalizations from §2 are inverted here, so statistics round-trip exactly (htdemucs.py:649).

What ONNX export changes: nothing learned

Everything above sits verbatim in the shipped .onnx graphs. The only surgery (tools/model-export/export_onnx.py, from PR #10) is on the two weight-free ends: torch.stft/istft use complex tensors, which ONNX lacks, so the STFT/iSTFT become matmuls against fixed sine/cosine tables and complex values ride as a trailing dimension of 2. Those baked DFT tables are the 136 MB "shared block" in the pipeline page's byte anatomy; the 42 M learned parameters here are the other 168 MB.
