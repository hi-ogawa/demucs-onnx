Demucs ONNX inference pipeline

The pipeline takes any song and returns its stems (§1), while the computation underneath is a single rigid, fixed-size black box (§2). This page is about the gap between those two facts: the inference flow that closes it (§3), the single pattern that flow repeats at three scales (§4), and the multiplicative bill for the repetition (§5).

1 · Split a song into stems

The job is music source separation, meaning the pipeline takes one mixed song and returns the same audio split into instrument stems, as if each part had been recorded alone. The rest of the page is how.

In

  • One song — a stereo wav file
  • Preprocessing aligns it with the format the models were trained on (stereo, 44.1 kHz)

Out

  • Stem wavs — raw f32, stereo, 44.1 kHz, stems sum to the mix
  • Full mode: drums / bass / other / vocals
  • Two-stems mode: <src> + no_<src>

2 · Run one fixed graph, swap only the weights

Everything §1 promises is more than the computation underneath can do. It is one computation, rigid on every axis. The input length is fixed, the output shape is fixed, and every run is stateless and deterministic. Its only degree of freedom is which weight set is plugged in.

The 4-stem axis of the output (sources, in upstream's parlance) is the separation itself, meaning the same 7.8 s four times over, as if played by each instrument group alone ([drums, bass, other, vocals]), ideally summing back to the input chunk.

The five weight sets specialize over that stem axis. The diagrams abbreviate the htdemucs_ft_* artifact names as ft_*. ft stands for fine-tuned, meaning each ft_* set is the generalist htdemucs trained further on one stem. That training sharpens that stem's output, but the fixed graph still emits all four. Which outputs are kept is the orchestration's call (§3):

Hosts appear at exactly one seam, which is the box itself. The graph runs through ONNX Runtime natively (CLI, Node) and ONNX Runtime Web in the browser, and the host swaps there while everything around the box stays shared. The graph's internals are out of scope here.

3 · Loop at three scales

This is how the gap between §1's promise and §2's box gets closed. Track-level conforming wraps three nested loops, each closing with its own average. The numbered rings are the same move at three scales, which §4 unpacks.

conform — channels to stereo, resample to 44.1 kHz
↓
normalize — whole track to zero-mean / unit-std of its mono fold-down (exact inverse at the end)
↓ normalized track
3per weight set — htdemucs: 1 · htdemucs_ft: 4 · ft + minus method: 1 (the target's specialist only)
2per shift — offset the track < 0.5 s, process whole, offset back
1per chunk — 343,980-sample window at stride 257,985
black box — session.run(chunk) → (1, 4, 2, 343980)
1average — triangular overlap-add → one full-length 4-stem estimate
2average — mean over the shift passes
3average — one-hot over weight sets, taking each stem from its specialist
↓ four stems (normalized domain)
denormalize — inverse of the track normalization
↓
stem arithmetic + write — full mode writes all four wavs, and two-stems mode follows §5

4 · Run many times, average

The box does one fixed-size computation, so every ring in §3 is the same move: vary something arbitrary about the input, run the box again, average the outputs. Total work is the scales multiplied: passes = models × shifts × chunks.

Chunks — vary the window position

5 · Pay per pass

Every pass is one full trip through the box, so the multiplication from §4 is the runtime you feel.

Config passes = models × shifts × chunks 10 s clip (2 chunks)
htdemucs 1 × 1 × 2 2 passes
htdemucs_ft 4 × 1 × 2 8 passes
htdemucs_ft + minus (bass) 1 × 1 × 2 2 passes, so fine-tuned bass comes at generalist cost
any, --shifts N (N ≥ 2) × N N-fold

One pass ≈ 5 s native / 9.5 s browser on the dev laptop.

Two-stems methods (bass example)

Method is plain arithmetic outside §4's pattern. It decides how no_bass is built, and it composes freely with model choice:

method add
no_bass = drums + other + vocals 		method minus
no_bass = mix − bass
htdemucs 1 model run 1 model run, with the same cost but different arithmetic
htdemucs_ft 4 model runs, each stem from its specialist 1 model run, because only the bass specialist's row of the §2 grid is needed
  • add: the backing is the estimated stems summed, so their artifacts stack
  • minus: the backing is the original mix minus the target estimate. With fine-tuned models, this requires running only the target specialist.
